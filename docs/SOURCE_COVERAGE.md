# Medical Source Coverage

本文定义公共版 Medical News Radar 的信源覆盖范围和接入标准。目标是稳定覆盖高价值医疗行业信号，而不是无限扩充来源数量。

## 分层模型

1. **阅读层**：首页默认展示筛选后的医疗信号，并保留全部、精选、热点、搜索、排序与多来源折叠。
2. **维护层**：`config/*.yml`、OPML、信源状态、探测脚本和 GitHub Actions。普通读者只在 `/sources.html` 查看只读状态。

## 信源优先级

| 优先级 | 来源 | 默认处理 |
| --- | --- | --- |
| 1 | 政府、监管机构、法院、WHO、FDA、NMPA、正式政策原文 | S级，优先接入公开 RSS/Atom |
| 2 | 权威医学期刊、研究机构、行业媒体、企业官方公告 | A级，使用公开 feed 或稳定公开页面 |
| 3 | 综合科技、财经与可靠转载媒体 | B级，限制条数并严格去重 |
| 4 | 聚合站、公众号、社交媒体观察源 | C级，只用于发现选题，默认谨慎接入 |

## 公共版支持

- 公开 RSS / Atom
- `config/sources.yml` 中的公开源
- 私有但不入库的 OPML 扩展
- 稳定、无需鉴权的 JSON API
- 少量结构稳定的公开列表页
- 静态 `data/*.json` 与 GitHub Pages 展示

当前配置以已经工作或经过验证的 WHO、CDC、FDA、NIH、NEJM、The Lancet、JAMA、BMJ、Nature Medicine、Medscape、Healthcare IT News、Fierce Healthcare 和 HIMSS 等公开来源为基础。新增失败或不稳定来源应先设 `enabled: false`。

## 不作为公共默认能力

- 需要登录、验证码、Cookie、浏览器会话或付费墙的页面
- 微信公众号完整正文抓取
- 私人邮箱正文和附件
- 原始社交时间线或脆弱第三方桥接
- 需要在前端保存 Token 的写操作
- 依赖云函数、数据库或长期运行服务的采集器

## 接入判定

候选信源应同时满足大部分条件：

- 医疗行业相关且能补足八大栏目中的明确空缺
- 来源身份和原始内容可核验
- 有稳定的 HTTP(S) 地址，优先 RSS/Atom
- 不需要登录、Cookie、验证码或绕过访问控制
- 有可用标题、时间和原文链接
- GitHub Actions 能以合理频率、体积和超时礼貌访问
- 不以转载、营销、养生偏方或未经证实内容为主

接入前运行 `scripts/source_probe.py`，核查 HTTP 状态、内容类型、feed 可解析性、更新时间、登录/拦截信号、医疗相关性、建议栏目与等级。探测通过只说明技术上可访问，不代表内容质量已获批准。

## 失败和淘汰策略

- 单次失败：保留，注册表显示 `warning` 或 `failed`，检查是否为临时故障。
- 持续失败或长期不更新：先将 `enabled` 设为 `false`，状态显示 `disabled`。
- 确认不再使用：保留一段时间的配置注释或 Git 历史后删除，避免立即硬删除导致误判。
- 任何来源失败都不应阻断其他来源、注册表或首页已有数据。

## 私有 OPML

将 `feeds/follow.example.opml` 复制为 `feeds/follow.opml`。真实文件已被 Git 忽略，只能保存在本地，或 Base64 编码后存入 `FOLLOW_OPML_B64` Secret。配置源和 OPML 源会合并，并按规范化 URL 去重。

## 安全与内容规则

- 只保存标题、来源、时间、链接、摘要/推荐理由和结构化元数据，不保存第三方完整正文。
- 不输出诊断或治疗建议。
- 不虚构政策文件号、生效日期、发布机构、临床结论、融资金额或审批结果。
- 政策和监管内容应优先链接原文；媒体转述与事实不一致时，以原文为准。
- DeepSeek 只做可选增强，未配置或请求失败时必须回退到确定性规则。
